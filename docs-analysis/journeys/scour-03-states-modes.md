# Scour 03 — App States / Modes & Transitions (Legacy VeloDrive)

Read-only enumeration of every distinct state/mode the legacy app (`/home/babymastodon/code/velo-drive/docs`) can be in, with behaviors, transitions, dark-mode specifics, and legacy code citations.

## How state is modeled

There is **no single state enum**. Observable states are combinations of engine flags plus a few mode strings and connection statuses, spread across:

- **Ride lifecycle** — `workout-engine.js` (pure logic, no DOM): booleans `workoutRunning`, `workoutPaused`, `workoutStarting` (`workout-engine.js:37-39`) + mode strings `freeRideMode` (`erg`|`resistance`, `:31`) and a vestigial `mode` always forced to `"workout"` (`:30, 636, 681-689`). UI is a pure render of `getViewModel()` (`workout-engine.js:574-602` → `workout.js:1090-1107`).
- **Connection** — `ble-manager.js`: per-device `state` string `connecting`|`connected`|`error` (no literal `disconnected`; disconnect maps to `error`). Emits `bikeStatus`/`hrStatus`/`hrBattery`.
- **Overlays** — independent DOM overlays each with their own open-flag and their own `keydown` Escape handler (welcome z60, planner z50, settings z46, picker z45, status z40).
- **Theme** — `:root` (auto via `@media prefers-color-scheme`), `:root.theme-light`, `:root.theme-dark` classes toggled by `theme-init.js`/`theme.js`.
- **Config / PWA / crash-recovery** — `storage.js` (IndexedDB `velo-drive`/`settings`), `welcome.js`, `service-worker.js`.

## E. Overlay / modal states & Escape behavior

**Picker and planner are two JS modules rendering into the SAME DOM overlay** (`#workoutPickerOverlay`/`#workoutPickerModal`), distinguished by `picker-mode` vs `planner-mode` classes — so they are mode-switched on one element, never truly stacked (schedule-mode logically layers picker over a `hideModal`-hidden planner). Settings is a separate overlay (`#settingsOverlay`). Each overlay registers its **own** `document`/`window` `keydown` Escape handler guarded by its open flag; there is **no central modal stack** — precedence emerges from per-handler guards plus the single ordered chain in the global handler `workout.js:1747-1772` (builder → picker-search → planner-detail → planner → picker). The **welcome handler `stopImmediatePropagation()`s** when open (z60, topmost) and the global handler short-circuits on `isWelcomeActive`, so welcome consumes Escape exclusively. Backdrops are a fixed `rgba(0,0,0,0.35)` in both themes.

**Escape stacking order (most-nested wins):** (1) welcome open → absorbs Esc exclusively. (2) builder → 1st Esc deselects block, 2nd → back to library (global handler defers via `isBuilderMode()`). (3) picker schedule-mode → closes picker, returns to planner calendar. (4) picker search focused → clears search (no close). (5) picker library → `picker.close()`. (6) planner detail → exits to calendar (one level; planner's own `exitDetailMode` and the global handler both target detail, never the whole planner). (7) planner calendar → `planner.close()` (planner's own handler refuses Esc here; close comes from the global handler). (8) settings logs → returns to settings main (genuine 2-level Esc stack in settings). (9) settings main → `closeSettings()`. (10) native confirm/alert → browser handles Esc (Cancel/dismiss), app handlers paused. Settings' Esc handler is **separate and uncoordinated** with the global picker/planner handler and does not `stopPropagation` — but the global handler's Esc block only branches on builder/picker/planner (never settings), so they don't conflict in practice. The cross-handler design (5 independent listeners, no central stack) is the main mis-handling risk surface.

## G. Theme system & DARK-MODE deep dive

**Mechanism:** `theme-init.js` (inline, pre-paint, anti-FOUC) reads `localStorage.themeMode` (`auto`/`light`/`dark`) and toggles `:root.theme-light`/`:root.theme-dark` (`theme-init.js:1-15`). `theme.js` `applyThemeMode`/`saveAndApplyThemeMode` mirrors to both localStorage and IndexedDB (`theme.js:6-33`; `storage.js:330-356`). Settings Auto/Light/Dark toggle (`settings.js:456-479`). **AUTO mode** = no class → `@media (prefers-color-scheme: dark)` block applies. So there are effectively three CSS sources of dark: `@media prefers-color-scheme` (auto), `:root.theme-dark` (forced).

**All overlays/components are variable-driven** — `welcome.css`, `settings.css`, `workout-picker.css`, `workout-planner.css` contain **no `:root.theme-dark` rules and (mostly) no `prefers-color-scheme` blocks**; they recolor purely via the variables defined in `workout-base.css`. Only `workout-base.css` and one `workout-planner.css` rule have explicit dark element rules.

### Exactly what changes in DARK mode

**Variable swaps (`:root.theme-dark` `workout-base.css:247-315`, mirrored in `@media` `:91-173`):**
- Surfaces collapse: `--bg`/`--surface`/`--surface-elevated`/`--nav-bg` all **#222222** (light: #f4f4f4). **All three surface vars are identical → modals don't visually separate from the page except by shadow.**
- Borders: `--border` #3a3a3a, `--border-subtle` #333 (light #d0d0d0/#e0e0e0).
- Text: `--text-main` #aaaaaa, `--text-numbers` #9b9b9b, `--text-muted` #777, `--stat-number-color` #f5f5f5 (light #666/#8d8d8d/#999/#333).
- Accent: `--accent` #ff5252 (light #e53935). CTA: `--cta-bg` #bd4d45 / hover #93463e / active #883f38 (light #d87d73/#ac554c/#9f4d45).
- Success/error: `--success` #6fdb9c, `--success-strong` #9ccc65, `--error` #ffb3ac, `--error-strong` #ff7043 (light greens/reds).
- Warning/alert: `--warning-text`/`--alert-text` #ffb86c on rgba(255,184,108,.12).
- Hover overlays flip from black-alpha to white-alpha: `--hover-strong` rgba(255,255,255,.1), `--hover-medium`/`--hover-light` rgba(255,255,255,.06), `--expanded-bg` rgba(255,255,255,.04).
- Tooltip: `--tooltip-bg` #303030, text #ddd, border rgba(255,255,255,.18).
- Chart: `--chart-empty-bg` rgba(34,34,34,.8), `--chart-empty-shadow` rgba(34,34,34,1), `--grid-line` #444, `--grid-line-subtle` #333.
- **Line colors differ (not just lightened):** `--power-line` #ffb300 (light #a607a6 — a *hue* change), `--hr-line` #e82210, `--cad-line` #6ad60b, `--ftp-line` #5c7ea6.
- **`--shade-bg` inverts: #ffffff in dark vs #000000 in light** — past-progress shade is white@5% in dark (lightens) vs black@5% in light (darkens) (`workout-chart.js:2021`).
- Zone colors slightly desaturated (`--zone-*`). Free-ride: `--freeride-fill` #3a3a3a / `--freeride-stripe` #4a4a4a (light #d8d8d8/#b0b0b0). Builder: `--wb-block-active-filter` brightness(1.4) saturate(.8), `--wb-insert-line` #6ba8ff.
- Planner palette: `--planner-selected-bg` #4d2e2c, `--planner-today-bg` #3d342b (light pink/cream).
- `--select-arrow` SVG stroke #bbbbbb (light #999).

**Explicit dark element rules (only in `workout-base.css` + one planner rule):**
- `a`/`a:hover` (`:364-372`) — recolor only.
- `.chart-tooltip` (`:486-493`) — bg #303030, text #ddd, border + heavier shadow rgba(0,0,0,.8).
- Nav/control/playback/clicktoggle hover backgrounds flipped to white-alpha (`:655-660, 666-670, 746-750, 808-812, 841-849`).
- `.debug-overlay`/`.debug-log`/`.debug-close-btn` (`:905-918`) — **legacy/unused**.
- `.status-overlay` (`:939-944`) — bg rgba(34,34,34,.8), text-shadow rgba(34,34,34,1).
- `.planner-day:hover` → `--hover-medium` (`workout-planner.css:548-552`).

### Dark-mode rendering risks (most likely to mis-render)

1. **Modal elevation lost** — `--surface-elevated` == `--bg` (#222) and box-shadow is a fixed dark `rgba(0,0,0,.35)`: picker (z45), settings (z46), planner (z50) panels barely separate from the dimmed-dark page. Backdrop is also dark-on-dark.
2. **Welcome illustration SVGs** (`img/trainer.svg`, `browser.svg`, `builder.svg`) use hardcoded hex incl. near-black `#1a1a1a`/`#4d4d4d` linework on the now-dark `--surface-elevated` → low contrast.
3. **Chart redraw dependency** — `workout-chart.js` reads colors via `getCssVar` at draw time; a theme switch without a redraw leaves stale line/shade colors. `--shade-bg` inversion is the starkest.
4. **Connection dots + battery-low color are hardcoded** (grey/amber/green/red/orange) and never theme-adjusted — fine on #222 but the color-only battery-low (no banner) is easy to miss.
5. **`theme-color` meta uses `prefers-color-scheme` media only** (`index.html:37-46`) — when a user **forces dark** while the OS is light (or vice-versa), the browser chrome/title-bar color won't match the forced theme.
6. **Native `confirm`/`alert` dialogs** render in OS theme, not the app's, in dark mode.
7. **`status-overlay` countdown/paused** depends on the theme class being applied before paint; the `@media` (auto) vs `.theme-dark` (forced) split means forced-dark-on-light-OS relies on the class path.

## Most likely mis-rendered / mis-handled

- **Dark mode:** (1) modal elevation lost on #222 surface + dark shadow (picker/settings/planner); (2) welcome SVGs with baked light-mode colors; (3) chart stale colors if not redrawn on theme switch (`--shade-bg` inversion); (4) `theme-color` meta only follows OS, mismatching forced theme; (5) native dialogs ignore app theme.
- **Modal Escape contexts:** Settings' Esc handler is uncoordinated with the main picker/planner handler (both can fire on one Escape if Settings overlays the picker). Welcome's `stopImmediatePropagation` masks all others. Builder-mode deliberately ignores Esc. Picker-search Esc clears search instead of closing.
- **Connection:** UI picker-cancel catches call `setBikeStatus("error")`/`setHrStatus("error")` with a bare string, so the dot reverts to grey instead of red (latent inconsistency).
