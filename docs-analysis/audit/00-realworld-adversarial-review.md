# Real-world logic — adversarial review & validity assessment

Two-wave multi-agent review of the logic the hermetic harness can't exercise
(Bluetooth, trainer control, sensor ingestion, workout auto-behaviors, lifecycle,
in-ride actions, persistence, audio/wake-lock). Coverage map:
`00-untested-realworld-coverage.md`.

- **Wave 1 (16 agents)** — per category, a *regression* agent (line-by-line
  legacy↔new diff) and an *edge-case* agent (poke holes in the new code). →
  **85 raw findings** (the agents self-rated 16 high / 20 medium / 26 low / 23 info).
- **Wave 2 (8 agents)** — a senior-engineer assessor per category re-opened the
  actual code and judged every finding: verdict, real severity, worth-fixing.

## Headline

Code-verification deflated the adversarial severities hard — the value of the
second wave:

| | valid | partially-valid | false-positive |
|---|---|---|---|
| **count** | 42 | 39 | 4 |

| assessed severity | high | medium | low | info | none |
|---|---|---|---|---|---|
| **count** | **1** | 14 | 42 | 21 | 7 |

Of the 16 self-rated "high" findings, **exactly one survived as high** after
reading the code; **4 were outright false positives**. **10 findings are
worth-fixing = yes**, almost all small, low-risk hardening. No data-corruption or
crash bug that fires under normal use was found — the gaps are flaky-hardware,

> Notable rejected scare: **AUTO-E2** ("auto-pause never fires when the trainer
> disconnects, ride keeps recording phantom power") — **false positive**. The
> transport's disconnect handler *does* emit a null/zero sample, so auto-pause
> triggers on a drop exactly as intended.

## My recommendation — what to actually fix

I endorse the assessors' code-grounded verdicts. Prioritized:

### Tier 1 — fix now (cheap, low-risk, genuinely real). ~1 focused PR.
1. **Async `tick()` re-entrancy + idempotent `endWorkout()`** — `ACT-E1` / `LIFE-E2`
   (and `AUTO-E1`). A BLE control-point write that stalls past 1 s lets two ticks
   overlap → double-count `elapsedSec`, **double-save the FIT**, fire
   `onWorkoutEnded` twice. Fix: a `tickInFlight` guard + an `ending` flag on
   `endWorkout`. Highest value here.
2. **Muting sound mid-countdown soft-locks the start** — `AUDIO-E1` (the only
   surviving **high**) + `AUDIO-R4`. Toggling Sound off during the 3-2-1 calls
   `beeper.stopAll()`, the countdown's `onDone` never fires, and the engine is
   stuck in `workoutStarting` forever (Start dead until reload). Fix:
   `setEnabled(false)` should stop only audio nodes, not tear down the countdown
   (+ a `beginRun` `if(!workoutStarting) return;` guard, `LIFE-E1`).
3. **ERG write reliability** — `ERG-E2` + `ERG-E4`. A swallowed GATT write
   failure still commits the dedupe state, so a failed target isn't retried for
   up to 10 s; a non-finite target coerces to **0 W** and busy-loops. Fix: only
   commit `lastErg*`/`lastErgSendTs` on a confirmed write; `return` on
   `!Number.isFinite(target)`.
4. **BLE state hardening** — `BLE-E1` (a true regression: the port hoisted
   `bikeControlPointChar = cpChar` *before* the stale-id guard, so a re-pair can
   leave ERG writes pointed at a dead GATT char — legacy committed it only after
   the guard) + `BLE-E2` (detach the disconnect handler before the synthetic
   `disconnect()` so a real drop can't consume the suppress-once flag).
5. **Auto-resume traps** — `ACT-E2` + `AUTO-E3`. After an auto-pause, resume
   needs `power ≥ 0.9 × target`; with a high manual ERG (free-ride) or a segment
   whose target rounds to 0 W, the rider physically can't hit it and is stuck on
   the pause overlay. Fix: floor the resume threshold / treat 0-or-free-ride
   targets as "resume on any positive power".
6. **Surface silent persist failures** — `PERS-E4`. Add a `.catch` (log/notify
   once) on the active-state persist so a structured-clone/quota failure isn't
   invisible. (Capping persisted `liveSamples` is a larger, separate change.)

### Tier 2 — worth doing, more judgment/effort
- **`AUDIO-E3` wake lock** (medium): genuinely unimplemented — the screen sleeps
  mid-ride. Real UX gap; `navigator.wakeLock` on ride start, re-acquire on
  visibility, release on pause/end. Bigger than Tier 1 but high user value.
- **`SENS-E1` FTMS resistance field width** (`index += 1` should be `+= 2` per
  spec): real for trainers that set the resistance flag — **but the same bug
  exists in legacy**, so a "fix" diverges from the parity oracle and may need the
  ftms self-test updated. Decide parity-vs-correctness before touching.
- **`PERS-E1` / `PERS-E2`** persistence hardening (cancel the debounce in
  `endWorkout`; disambiguate same-second FIT filenames): low real-world odds,
  cheap if touched.

### Rejected (false positives — no action)
`AUTO-E2` (disconnect already emits null → auto-pause fires), `BLE-E6`
(consumer already guards bike-vs-strap HR), `LIFE-E4` (`segDurationSec` floors at
1 s, so `workoutTotalSec` can't be 0), `AUDIO-R7` (deferred `onDone` is documented
and correct).

**Bottom line:** the new real-world code is sound; nothing here blocks the live
deploy. Tier 1 is a single small, well-contained hardening PR I'd recommend
landing. Say the word and I'll implement it (each fix is revert-verifiable).

## C1. BLE device lifecycle & connectivity

> The BLE lifecycle port is a faithful, mostly-correct translation of legacy/ble-manager.js. #### [BLE-E1] Stale-device teardown leaves bikeControlPointChar pointing at a dead GATT
*Edge-case (new code) · original **HIGH** → assessed **MEDIUM** · verdict **valid** (confidence high) · worth fixing: **yes***

- **Location:** `web/src/ports/web/WebBluetoothTransport.ts:416-433`
- **Finding:** During a bike re-pair, connectToBike sets this.bikeControlPointChar = cpChar (line 416) BEFORE the desired-id check. If the user re-pairs to a different bike while a connect is in flight (bikeDesiredDeviceId changes mid-await), the original connect resumes, finds deviceId !== bikeDesiredDeviceId (line 425), disconnects the server and returns — but never nulls bikeControlPointChar. It now references the just-disconnected device's control point. setTrainerState only guards on this.bikeConnected && this.bikeControlPointChar (line 702); both can be truthy if the NEW device's connect already completed and set bikeConnected=true, while bikeControlPointChar still points at the OLD dead char. ERG/resistance setpoint writes then go to a disconnected/foreign GATT object, either throwing (caught+logged, rider gets no ERG) or no-op'ing. Also, no gattserverdisconnected handler is attached for the stale device (handler is added at line 448, after the return), so the silent server.disconnect() at line 428 doesn't trigger any cleanup either.
- **Evidence:** Line 416: `this.bikeControlPointChar = cpChar;` runs before line 425 `if (deviceId !== this.bikeDesiredDeviceId) { ... server.disconnect(); return; }` — the early return never clears bikeControlPointChar. setTrainerState line 702: `if (!this.bikeConnected || !this.bikeControlPointChar) return;`
- **Assessment:** Real bug. Line 416 `this.bikeControlPointChar = cpChar;` runs BEFORE the stale-id check at line 425, and the early-return teardown branch (lines 426-432) calls server.disconnect() but never nulls bikeControlPointChar. setTrainerState guards only `if (!this.bikeConnected || !this.bikeControlPointChar) return;` (line 702). If a re-pair to a different bike is in flight and the OLD connect resumes post-await to find itself stale, it disconnects but leaves bikeControlPointChar pointing at the dead char; if the NEW device's connect has set bikeConnected=true, ERG writes go to sendErgSetpointRaw which uses this.bikeControlPointChar (line 677) — the wrong/dead char. The write would throw and be swallowed (line 680), so the rider silently gets no ERG. Legacy had the SAME structure (controlPointChar is local until committed at line 745, so legacy actually does NOT have this bug — legacy commits bikeState.controlPointChar only after the stale check). So this is a genuine NEW regression: the port hoisted the assignment before the guard. Severity adjusted to medium because the trigger requires a precise mid-await re-pair to a different device, but the consequence (silent ERG failure) is real and matches the legacy-safe design intent.
- **Recommendation:** Fix now. Move `this.bikeControlPointChar = cpChar;` into the commit block after the stale-id check (after line 433), mirroring legacy which only assigns bikeState.controlPointChar at commit time. Apply the same to connectToHr if any equivalent char is hoisted.

#### [BLE-E2] suppressAutoReconnectOnce consumed by ANY disconnect, not the intended manual one
*Edge-case (new code) · original **HIGH** → assessed **MEDIUM** · verdict **valid** (confidence medium) · worth fixing: **yes***

- **Location:** `web/src/ports/web/WebBluetoothTransport.ts:319-325, 490-496`
- **Finding:** connectBikeViaPicker sets bikeSuppressAutoReconnectOnce=true then calls bikeServer.disconnect() to tear down the old connection on a cancelled re-pair (lines 319-325). The flag is meant to swallow exactly that one synthetic disconnect. But the flag is a plain boolean consumed by whichever gattserverdisconnected fires first. Real-world trigger: the bike drops on its own (battery/RF) at almost the same moment the user opens the picker and cancels. The hardware disconnect fires onBikeDisconnect first, consumes the suppress-once flag (line 493) and skips scheduling reconnect; then the manual server.disconnect() fires a second disconnect which now DOES schedule a reconnect — or vice versa. Either way the user can end up with auto-reconnect permanently skipped for a genuine drop, leaving the bike disconnected mid-ride with only an 'disconnected' status and no retry.
- **Evidence:** Line 320: `this.bikeSuppressAutoReconnectOnce = true;` then line 322 `this.bikeServer.disconnect();`. onBikeDisconnect line 490-495: `if (!this.bikeSuppressAutoReconnectOnce) { this.scheduleBikeAutoReconnect(true); } else { this.bikeSuppressAutoReconnectOnce = false; ... }` — the flag is global, not bound to a specific disconnect event/device.
- **Assessment:** Real design weakness, and legacy shares it. connectBikeViaPicker line 320 sets bikeSuppressAutoReconnectOnce=true then line 322 calls bikeServer.disconnect(). The flag is a plain boolean consumed by whichever gattserverdisconnected handler fires (onBikeDisconnect line 490-495). If a genuine hardware drop races the synthetic manual disconnect, the real drop can consume the suppress flag (skipping its legitimate reconnect) while the synthetic one schedules a reconnect — net effect a genuine drop gets no auto-reconnect that session. The window is small (both disconnects must fire near-simultaneously) and requires the user to cancel a re-pair picker at the same instant the bike drops, so 'permanently disabled mid-ride' is somewhat overstated — onBikeDisconnect always re-arms scheduleBikeAutoReconnect on the NEXT real drop. Legacy/ble-manager.js lines 1022-1024 has identical logic, so it is not a regression but a faithfully-ported latent bug. Worth hardening.
- **Recommendation:** Fix now (cheap hardening): detach the disconnect handler before calling the synthetic server.disconnect() so the manual teardown produces no handler invocation, eliminating the race entirely. This is strictly better than legacy.

#### [BLE-R1] Auto-reconnect of a stale device now flips status to connecting (legacy guarded it)
*Regression (legacy↔new) · original **MEDIUM** → assessed **LOW** · verdict **valid** (confidence high) · worth fixing: **maybe***

- **Location:** `web/src/ports/web/WebBluetoothTransport.ts:389, 545`  ·  legacy `legacy/ble-manager.js:598-600, 789-791`
- **Finding:** In legacy connectToBike/connectToHr the initial "Connecting to X…" status is emitted ONLY when the device being connected is still the desired device (deviceId === bikeDesiredDeviceId). The new code emits updateBikeStatus('connecting') / updateHrStatus('connecting') unconditionally at the top of connectToBike/connectToHr. Trigger: an in-flight auto-reconnect (or a connect attempt for a device the user has since re-paired away from) will now overwrite the status dot/message of the currently-desired device with a 'connecting' state for a device the user no longer cares about. The status surface for the active device gets clobbered.
- **Evidence:** Legacy (bike): line 598-600 `if (deviceId === bikeDesiredDeviceId) { updateBikeStatus("connecting", ...); }` and HR line 789-791 `if (deviceId === hrDesiredDeviceId) { updateHrStatus("connecting", ...); }`. New (bike): line 389 `this.updateBikeStatus('connecting');` with no deviceId guard; New (HR): line 545 `this.updateHrStatus('connecting');` with no guard.
- **Assessment:** Confirmed divergence. Legacy connectToBike line 598-600 emits the initial connecting status only `if (deviceId === bikeDesiredDeviceId)`; HR line 789-791 mirrors it. New connectToBike line 389 does `this.updateBikeStatus('connecting');` unconditionally (HR line 545 likewise), with no deviceId guard. So a connect for a no-longer-desired device clobbers the active device's status surface. However the trigger is narrow: connectToBike is only invoked from the picker (desired id was just set to this device), the boot reconnect (desired matches), and the backoff timer (which re-reads the current desired id at line 264 before connecting). The only way to reach connectToBike with a stale id is a re-pair landing mid-await between timer-fire and gatt.connect — a sub-millisecond window. Real but low impact and cosmetic (status dot only).
- **Recommendation:** Defer or fix cheaply: wrap the initial updateX('connecting') in `if (deviceId === this.bikeDesiredDeviceId)` to mirror legacy. Low risk, restores parity, but not user-urgent.

#### [BLE-R2] Dropped isAuto early-skip: a fired backoff timer connects a no-longer-desired device
*Regression (legacy↔new) · original **MEDIUM** → assessed **LOW** · verdict **partially-valid** (confidence high) · worth fixing: **maybe***

- **Evidence:** Legacy (bike) lines 591-596: `if (isAuto && desiredAtStart && desiredAtStart !== deviceId) { log(...); return; }`; HR lines 782-787 same. #### [BLE-E3] Auto-reconnect attempt emits 'connecting', regressing the error dot and discarding the retry countdown
*Edge-case (new code) · original **MEDIUM** → assessed **LOW** · verdict **partially-valid** (confidence high) · worth fixing: **maybe***

- **Location:** `web/src/ports/web/WebBluetoothTransport.ts:389, 257-272`
- **Finding:** scheduleBikeAutoReconnect sets status to 'error' with a 'Will retry in Ns…' message (line 257-260). When the timer fires it calls connectToBike, which immediately calls updateBikeStatus('connecting') (line 389). If the device is out of range, gatt.connect() can hang for the platform timeout (often 10-30s) before rejecting. During that whole window the UI shows 'connecting' (typically an amber/idle dot), not the error/retry state, and the user has no indication the link is actually down and backing off. On repeated failures the dot flickers error→connecting→error every cycle. With backoff capped at 10s but connect timeouts longer than 10s, the perceived state is mostly 'connecting' even though nothing is connected.
- **Evidence:** scheduleBikeAutoReconnect line 257 `this.updateBikeStatus('error', "...Will retry in Xs…")`; the timer callback line 267 calls `this.connectToBike(dev)` which at line 389 unconditionally does `this.updateBikeStatus('connecting');`.
- **Assessment:** The mechanism is real: scheduleBikeAutoReconnect sets error+countdown (line 257), then the timer fires connectToBike which immediately sets 'connecting' (line 389), and if gatt.connect() hangs for the platform timeout the dot shows connecting for that whole window. This is essentially a duplicate of BLE-R1's root cause (unconditional 'connecting' emit). BUT legacy did the SAME thing in the timer callback: connectToBike(dev,{isAuto:true}) at legacy line 255 calls updateBikeStatus('connecting',...) at legacy line 599/609 (since deviceId === desired during a reconnect). So during the GATT-connect window legacy ALSO showed 'connecting', not 'error'. This is therefore NOT a regression — it matches legacy. The error→connecting→error flicker on repeated failures exists in legacy too. Severity downgraded; it is an inherited cosmetic quirk, not a new defect.
- **Recommendation:** Defer. If the flicker is annoying, suppress the 'connecting' status for backoff-triggered connects (would require the isAuto flag back); otherwise wontfix as it matches the shipped legacy behavior.

#### [BLE-E4] ERG throttle commits lastTrainerMode even when the BLE write fails
*Edge-case (new code) · original **MEDIUM** → assessed **LOW** · verdict **partially-valid** (confidence high) · worth fixing: **maybe***

- **Location:** `web/src/ports/web/WebBluetoothTransport.ts:673-682, 705-717`
- **Finding:** setTrainerState updates lastTrainerMode/lastErgTargetSent/lastErgSendTs (lines 715-716) AFTER awaiting sendErgSetpointRaw, but sendErgSetpointRaw swallows write errors internally (line 679-681, catch logs and returns). So a transiently-failed setpoint write is recorded as 'sent': lastErgTargetSent now equals target and lastErgSendTs is updated, so the SAME target won't be re-sent for another 10s even though the trainer never received it. Real-world trigger: a single dropped GATT write right after an interval power step — the trainer stays at the previous wattage for up to 10s while the app believes the new target is set. The engine's own re-send heuristic also keys off unchanged target, so it won't correct it either.
- **Evidence:** sendErgSetpointRaw line 679-681: `} catch (err) { this.log('Failed to set ERG target: ' + err); }` (no rethrow). setTrainerState line 712-717: `if (needsSend) { await this.sendErgSetpointRaw(target); this.lastTrainerMode='erg'; this.lastErgTargetSent=target; this.lastErgSendTs=tNow; }` — the bookkeeping commits regardless of whether the write actually succeeded.
- **Assessment:** The bookkeeping-commits-on-failed-write behavior is real: sendErgSetpointRaw swallows the write error (lines 679-681, no rethrow) and setTrainerState commits lastErgTargetSent/lastErgSendTs unconditionally after the await (lines 713-716). A single dropped GATT write records the target as sent. HOWEVER: (1) this is an EXACT port of legacy — legacy sendErgSetpointRaw swallows errors (lines 486-488) and setTrainerStateInternal commits after await (lines 520-523), so not a regression; (2) the 'won't re-send for 10s' claim is wrong about the worst case: the engine's tick() calls sendTrainerState(false) every tick, and the throttle re-sends whenever `tNow - lastErgSendTs >= 10s` OR the target changes. So a stuck wattage self-corrects within at most ~10s on a stable target, and IMMEDIATELY on the next interval step (target change resets the needsSend predicate via lastErgTargetSent !== target — but since the failed write DID set lastErgTargetSent=target, only an unchanged target is stuck). The 'performance.now wall-independent clock' claim in the title is a non-issue (nowSec is only used for relative deltas). Net: real minor robustness gap, faithfully inherited, bounded to ~10s on a steady-state target.
- **Recommendation:** Maybe fix as an improvement over legacy: have sendErgSetpointRaw return success/throw and only commit lastErgTargetSent/lastErgSendTs on confirmed write, so a failed write retries next tick (~1s) instead of up to 10s. Low risk, clear win, but not a regression so not urgent.

#### [BLE-E5] Concurrent in-flight reconnect: two gatt.connect() race on the same device, duplicating notification listeners
*Edge-case (new code) · original **MEDIUM** → assessed **LOW** · verdict **partially-valid** (confidence medium) · worth fixing: **maybe***

- **Location:** `web/src/ports/web/WebBluetoothTransport.ts:251, 261-272, 385-452`
- **Finding:** There is no in-flight guard on connectToBike. Sequence: backoff timer fires (line 261), clears bikeReconnectTimerId, and calls connectToBike(dev) which awaits gatt.connect(). While that is pending, the device's prior gattserverdisconnected can still arrive (or onBikeDisconnect from a stale server) and call scheduleBikeAutoReconnect(true), which sees bikeConnected===false and !bikeReconnectTimerId, so it schedules ANOTHER timer. When that second timer fires you now have two overlapping connectToBike attempts on the same BtDevice, each adding its own characteristicvaluechanged + gattserverdisconnected listeners (lines 398,411,448). Duplicate notification listeners mean parseIndoorBikeData runs twice per packet (double bikeSample emits), and the second connect overwrites bikeServer/bikeControlPointChar, orphaning the first.
- **Evidence:** scheduleBikeAutoReconnect guard line 251 only checks `if (!device || this.bikeConnected) return;` — it does NOT check whether a connect is already in flight or a timer fired. connectToBike has no re-entrancy guard and unconditionally adds listeners at lines 398, 411, 448.
- **Assessment:** There is genuinely no in-flight/re-entrancy guard on connectToBike, and scheduleBikeAutoReconnect line 251 only checks `!device || this.bikeConnected`. The legacy code explicitly documents 'Multiple connect calls for the same device are allowed to run in parallel' (legacy comment line 580) — so concurrency was a KNOWN accepted property, not a regression. The duplicate-listener concern is real in principle (lines 398/411 add characteristicvaluechanged each connect, and only the gattserverdisconnected handler is dedup-detached at line 439), so two overlapping successful connects on one device could double-emit bikeSample. BUT reaching two overlapping connects requires: timer fires connectToBike (which clears the timer id at 262) AND a stale gattserverdisconnected schedules a new timer that also fires before the first connect resolves — and crucially scheduleBikeAutoReconnect's `if (this.bikeConnected) return` plus the single-timer cancelBikeAutoReconnect at line 255 collapse most double-schedules. The window is narrow and the symptom (transient double bikeSample) is benign for the engine (it just overwrites lastSamplePower). Real but low impact and matches legacy's accepted-parallelism stance.
- **Recommendation:** Maybe: add an in-flight boolean guard to connectToBike/Hr (improves on legacy and prevents duplicate notification listeners). Defer if no double-sample symptom is observed.

#### [BLE-R3] Boot reconnect connects immediately instead of scheduling, skipping legacy backoff-status path
*Regression (legacy↔new) · original **LOW** → assessed **LOW** · verdict **valid** (confidence high) · worth fixing: **no***

- **Location:** `web/src/ports/web/WebBluetoothTransport.ts:203-231`  ·  legacy `legacy/ble-manager.js:963-985`
- **Finding:** Legacy maybeReconnectSavedDevicesOnLoad sets the desired ID and calls scheduleBikeAutoReconnect(true)/scheduleHrAutoReconnect(true), which emits a 'Device disconnected. Will retry in 1s…' status and connects after the 1s timer. The new maybeReconnectSavedDevices instead calls connectToBike/connectToHr immediately (emitting 'connecting' right away) and only falls back to scheduleX(true) on failure. Functionally both end up connected, but on boot the new code skips the 1s delay and the initial retry-status message, and immediately emits a 'connecting' status (compounded by finding #1's unconditional connecting emit). #### [ERG-E2] Swallowed BLE write failure still records dedupe state; failed target not retried for up to 10s
*Edge-case (new code) · original **HIGH** → assessed **MEDIUM** · verdict **valid** (confidence high) · worth fixing: **yes***

- **Location:** `web/src/ports/web/WebBluetoothTransport.ts:673-682, 712-717`
- **Finding:** sendErgSetpointRaw/sendResistanceLevelRaw catch and swallow the write rejection (only logging). setTrainerState updates lastErgTargetSent/lastErgSendTs AFTER the await regardless of whether the write actually succeeded. The next ticks with the same (correct) target are then deduped out, and the trainer keeps the OLD target until either 10s elapse from the failed-send timestamp or the target changes again. - **Evidence:** sendErgSetpointRaw (l.679-681) swallows: 'catch (err) { this.log(\'Failed to set ERG target: \' + err); }'. setTrainerState (l.712-717) updates after await unconditionally: 'await this.sendErgSetpointRaw(target); ... this.lastErgTargetSent = target; this.lastErgSendTs = tNow;'.
- **Assessment:** Real and correctly described. sendErgSetpointRaw catches and only logs the rejection (l.679-681 `catch (err) { this.log('Failed to set ERG target: ' + err); }`), returning normally. setTrainerState then commits dedupe state unconditionally after the await (l.713-716): `await this.sendErgSetpointRaw(target); this.lastTrainerMode='erg'; this.lastErgTargetSent=target; this.lastErgSendTs=tNow;`. Severity adjusted to medium rather than high: it self-heals within ~10s and the trigger (write rejection exactly on the change tick) is intermittent — but at an interval transition that is a meaningful wrong-resistance window. Note: legacy has the identical swallow+commit structure (ble-manager.js:486-488, 520-523), so this is also not a regression — but it is a genuine latent bug worth fixing in the new code.
- **Recommendation:** Fix now. Have sendErgSetpointRaw/sendResistanceLevelRaw return success (or rethrow), and only commit lastErg*/lastResistance*/*SendTs when the write resolved — so the next tick retries. Bundle with E4.

#### [ERG-E4] NaN/non-finite ERG target silently coerced to 0 W instead of rejected
*Edge-case (new code) · original **MEDIUM** → assessed **LOW** · verdict **valid** (confidence medium) · worth fixing: **yes***

- **Location:** `web/src/ports/web/WebBluetoothTransport.ts:706-716, 675`
- **Finding:** If state.value is NaN (e.g. a corrupted restored manualErgTarget, or an FTP-derived target where currentFtp became NaN before the Number.isFinite guards catch it), Math.round(NaN)=NaN. #### [ERG-E3] Dedupe committed after await allows duplicate concurrent writes / lost-update ordering
*Edge-case (new code) · original **MEDIUM** → assessed **LOW** · verdict **partially-valid** (confidence medium) · worth fixing: **maybe***

#### [ACT-E1] async re-entrant tick() can double-fire endWorkout / double-save FIT
*Edge-case (new code) · original **HIGH** → assessed **MEDIUM** · verdict **valid** (confidence high) · worth fixing: **yes***

#### [PERS-E4] saveActiveState writes raw liveSamples every debounce; structured-clone/quota failure silently loses active state
*Edge-case (new code) · original **MEDIUM** → assessed **MEDIUM** · verdict **valid** (confidence high) · worth fixing: **yes***

- **Location:** `web/src/core/engine.ts:283-305, web/src/ports/web/WebFileStore.ts:227-244`

If any field is non-clone-safe or the value grows past quota, the put rejects and saveActiveState's promise rejects — but tick() calls it via `void this.persistActiveState()` with no catch, so the rejection is an unhandled promise and the active-state snapshot silently stops updating. - **Evidence:** buildActiveSnapshot includes `liveSamples: this.liveSamples` (engine.ts:295); persistActiveState `return this.fileStore.saveActiveState(this.buildActiveSnapshot());` (engine.ts:279-281) and tick fires it via `void this.persistActiveState()` (engine.ts:275) / scheduleSaveActiveState with no error handling. setSetting clones+puts the entire value (WebFileStore.ts:237-243).

#### [PERS-E1] Pending debounced saveActiveState can fire AFTER endWorkout's persistIdleState, resurrecting a finished ride on reload
*Edge-case (new code) · original **HIGH** → assessed **LOW** · verdict **partially-valid** (confidence high) · worth fixing: **maybe***

- **Location:** `web/src/core/engine.ts:271-277,716`
- **Finding:** endWorkout() never cancels the pending 500ms debounce timer armed by scheduleSaveActiveState(). The last tick before the ride completes (or any setMode/setFtp/adjust* call within 500ms of finishing) arms saveStateTimer to run persistActiveState() in 500ms. endWorkout() runs synchronously-ish, resets all state, and calls persistIdleState() — but the still-pending timer then fires and overwrites activeWorkoutState with buildActiveSnapshot(). Because buildActiveSnapshot() captures `this` LIVE at fire time (after endWorkout zeroed everything) the snapshot is mostly idle, but workoutRunning is read live too — the risk is the inverse ordering: if the auto-completion endWorkout() is triggered from inside tick() while a separate scheduleSaveActiveState() from an earlier tick is still pending, the pending save races persistIdleState. More concretely, on a normal finish the elapsedSec>=workoutTotalSec branch calls endWorkout() and returns, but scheduleSaveActiveState() was already called earlier in that same tick (line 496) BEFORE the end check (line 487)? No — end check precedes it. The reachable trigger: user hits a control (setFtp/adjustManualErg) at second ~T, arming the timer; ride auto-completes within the next 500ms; persistIdleState writes idle; then the debounced persistActiveState writes a snapshot with workoutRunning=false but elapsedSec/liveSamples already cleared — harmless. The genuinely harmful path is the opposite: a debounced save scheduled while running fires, then nothing clears it, leaving a stale 'running' activeState if endWorkout's persistIdleState write loses the IDB race (both are fire-and-forget `void`, no ordering guarantee between the two un-awaited transactions).
- **Evidence:** scheduleSaveActiveState: `if (this.saveStateTimer) return; this.saveStateTimer = this._setTimeout(() => { this.saveStateTimer = null; void this.persistActiveState(); }, 500);` (engine.ts:271-277). endWorkout never touches saveStateTimer and fires `void this.persistIdleState();` (engine.ts:716) with no await/ordering vs the pending debounced `void this.persistActiveState()`. - **Assessment:** The race is real in principle but the 'high / resurrect a finished ride' framing is overstated, and the finding's own prose talks itself down. On the normal-finish path the end check (engine.ts:487 `elapsedSec >= workoutTotalSec` -> await endWorkout(); return) runs BEFORE scheduleSaveActiveState() (496) within the same tick, so a finishing tick does not arm a new timer. A timer armed by an EARLIER tick can still be pending when endWorkout runs; if it fires after persistIdleState, persistActiveState() calls buildActiveSnapshot() which reads `this` LIVE — and endWorkout already set workoutRunning=false, elapsedSec=0, liveSamples=[] (703-708) — so the late snapshot is an idle-equivalent state, NOT a running ride. The only genuinely bad outcome requires the two un-awaited IDB puts (`void persistIdleState` vs `void persistActiveState`) to complete out of submission order; in IndexedDB, two readwrite transactions on the same store run in the order they were requested, so the idle write (requested last, from endWorkout) lands last in practice. This makes resurrection effectively unreachable in the real browser. Duplicate of PERS-R1.
- **Recommendation:** Defer. A one-line `if (this.saveStateTimer) { this._clearTimeout(this.saveStateTimer); this.saveStateTimer = null; }` at the top of endWorkout is cheap, correct hardening and removes the reasoning burden — fix it opportunistically, but it is not a high-severity bug and diverges (harmlessly) from legacy.

## C8. Audio, wake lock & backgrounding

> The audio/overlay port (beeper.ts) is a deliberately 'focused' reimplementation — its header documents that it ports only the start countdown + a basic cue primitive — so several findings are real-but-intentional fidelity reductions (R2 siren→single beep, R3 dropped cue cadence, R5 reflow, R6 font ratio) that are correctly flagged but not worth restoring unless full audio fidelity is a goal. Of 14 findings: 7 valid, 5 partially-valid, 1 false-positive (R7, self-acknowledged), and several edgecases (E4/E5/E6) are faithful ports of legacy behavior rather than regressions. The wake-lock gap (E3) and AudioContext-resume gap (R1/E2, overlapping) are legitimate missing-resilience items worth implementing but lower priority. Background-tab tick drift (E6) is a documented, deliberately-deferred known limitation.
