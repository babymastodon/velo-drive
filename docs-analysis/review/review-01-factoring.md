# Review 01 — Overall Architecture & Factoring

**Scope:** `web/src` (the Svelte rewrite). **Lens:** module structure, boundaries, encapsulation, right-sized abstraction.
**Goal being measured against:** lightweight, simple, straightforward, well-encapsulated — explicitly NOT a heavy framework or deep object model. Zero runtime deps today (preserve that).
**Method:** read the actual code (every file in `core/ ui/ state/ ports/ app/`); mapped the full import DAG; cross-checked the port interfaces against their concrete impls and against actual UI call sites.
**Constraint noted throughout:** `engine.ts` behavior is pinned by tests and the re-hosted CSS is verbatim — every fix below must preserve behavior. **This review changes no code.**

LOC by layer: core 8,758 · ui 6,822 · ports 1,729 · state 378 · app 365 (40 files, ~18.1k total).

---

## Structural strengths

These are genuinely clean and should survive the merge intact:

- **The layered DAG is real and (almost) acyclic.** `core/` imports *nothing* upward — verified: zero references to `ui/`, `state/`, or `app/` anywhere in `core/`. The only upward-looking edge is `engine.ts` importing the **port interfaces** `TrainerTransport`/`FileStore` (type-only, dependency-injected) — which is exactly the intended seam, not a leak. Intra-`core` edges form a clean tree (`model` ← everything; `metrics` ← `planner-analysis`, `chart`, `engine`; no cycles).
- **The engine is a model citizen.** `core/engine.ts` (948 lines) is a DOM-free state machine with every platform dependency injected (`transport, fileStore, beeper, now, setInterval, …`), a flat private-field state, and a single `getViewModel()` projection. No framework, no DOM, no globals captured at import. This is the spine of the app and it's done right.
- **State layer is thin and signal-only.** `state/*.svelte.ts` (378 LOC total) holds only `$state` view-models and dispatch — `EngineStore` is 20 lines, `LogsStore` 25, `DialogStore` is a clean promise-backed modal. No business logic leaked in. `dialog.svelte.ts` replacing native `alert/confirm/prompt` with a promise API is a tidy abstraction.
- **The codecs are cohesive and pure.** `zwo.ts` (1303) and `fit.ts` (1305) are single-purpose format codecs, DOM-free, importing only `model`. Long is *correct* for codecs — do not split them.
- **Pure-helper extraction in UI is good.** Presentation logic that *could* have rotted inside `.svelte` files was instead pulled into testable pure TS modules: `hud-format.ts`, `hud-coaching.ts`, `welcome-scene.ts`, `compat.ts`, `theme.ts`. Components stay declarative.
- **Composition root is explicit and small.** `app/app.ts` (98 lines) wires ports → engine → stores in one readable place; `shim.ts` isolates all test-harness platform swapping behind a single boot call. Zero runtime deps preserved.
- **Reusable modal chrome.** `OverlayModal.svelte` is shared by Settings + Dialog (backdrop-gesture logic written once). Good small abstraction, not over-engineered.

## Biggest 3 wins

1. **Make the FileStore port tell the truth (#1).** Promote the history/schedule surface + its domain types onto the `FileStore` interface and retype the three views to it. Purely mechanical, zero behavior change, and it converts a *claimed* port boundary into a *real* one — the single highest-leverage structural fix. **No-brainer.**
2. **Pull domain logic out of the persistence adapter (#2).** Move ride-analytics (`buildPreview`) and schedule rules into pure `core/history.ts` + `core/schedule.ts`, and relocate the domain types to `core/model.ts`. Behavior-preserving move that cleanly separates I/O from domain and gives the planner logic a testable home. Pairs naturally with #1.
3. **De-duplicate `chart.ts` via a shared primitives module (#3).** Extract the color/zone/pattern helpers + one `renderSegmentPolygon` + one grid/FTP-line drawer; have all five renderers call them. Kills the only real copy-paste left in `core/` while leaving renderer bodies (and thus the pixel-diff/verbatim-CSS behavior) untouched.

**Two explicit "leave it alone" calls:** `builder-backend.ts` (2095) and the `zwo.ts`/`fit.ts` codecs are cohesive and correctly sized — do not split them for line-count reasons. And the `chart.ts` → `document` CSS-var coupling (#4) is a deliberate, behavior-pinned choice; flag it as known, don't "fix" it now.
