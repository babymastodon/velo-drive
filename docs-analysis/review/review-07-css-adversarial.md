# Review 07 — Adversarial CSS Review (read-only)

Scope: `web/src/styles/*.css` — `workout-picker.css` (1349), `workout-base.css` (1190), `workout-planner.css` (583), `welcome.css` (495), `settings.css` (421). Bundled via `@import` from `index.css:5-9`.

Method: every claim below is cited to a real `file:line`. Counts were measured with `grep`/`sed`, not eyeballed. This is a hostile read; severities are mine.

## Measured baseline (the numbers the rest of the review leans on)

- **235 raw hex literals**, of which **234 live in `workout-base.css`** (the token file) and **1 in `welcome.css:12`**. The other three files (`picker`, `planner`, `settings`) are hex-clean — they go through tokens. So the entire "magic color" problem is concentrated in the one file that is supposed to be the single source of truth, and it both *defines* tokens and *bypasses* them.
- **`!important`: 11 total** — 6 in `workout-picker.css`, 3 in `workout-planner.css`, 1 in `welcome.css:435`, 1 in `workout-picker.css:1347`. Zero in the token file (good) but every behavioral file uses them to win visibility fights.
- **Dark color is defined in TWO mechanisms that must be hand-synced:** `@media (prefers-color-scheme: dark)` AND `:root.theme-dark`. In `workout-base.css` the two dark blocks each redefine **70 distinct tokens** (`:root.theme-dark` lines 263-343 vs `@media` lines 92-176) — a near-verbatim copy. Light is *also* tripled: base `:root` (3-90), `:root.theme-light` (178-261), plus the implicit light. That's roughly **3x the token surface** for every value.
- **z-index is an unscaled grab-bag:** values seen are `1, 2, 3, 10, 40, 45, 46, 50, 60, 61` across files (see table #9). No named scale, no shared variable; modal stacking order is encoded as bare magic ints scattered over 5 files.

## The 5 worst things in this CSS (ranked by real damage)

### 1. The dark palette is hand-maintained in two parallel blocks (70 tokens × 2) — Finding #1, #2, #3
`workout-base.css:92-176` (`@media` dark) and `:263-343` (`:root.theme-dark`) are a 70-token verbatim duplicate, and the file is littered with comment-tombstones (L249-253, L256-260, L523-533, L1030-1031, L1071-1077) each documenting a real bug caused when one copy was updated and the other wasn't. This is not a hypothetical drift risk — **it has already shipped wrong colors multiple times.** Worse, #2 shows a token duplicated a *third* time inside one block (L154-155 vs L158-159) with formatter-breaking indentation. Every future color change is a multi-site edit with no lint catching a miss. This single design choice generates most of the file's other problems (#3, #6's tooltip triplication, #8's dead-block re-theming).

### 2. The token file bypasses its own tokens — Finding #6, #13
234 of the 235 raw hex sit in the file whose entire job is to *be* the token source. `.chart-tooltip` hardcodes `#ffffff/#666666/...` four times (base:505,515,528,535) while `--tooltip-*` tokens (L31-33) sit unused two screens up and a sibling `.picker-tooltip` *does* use them — so the two tooltips are guaranteed to diverge. The hover literal `rgba(0,0,0,0.06)` is pasted 12 times next to the `--hover-strong` token that equals it. The abstraction exists and is actively ignored, which is worse than having no tokens: it gives false confidence that changing a token changes the UI.

### 3. `!important` arms race built on over-qualified selectors — Finding #5, #10
`.picker-search-clear:hover` triple-`!important`s background/border (picker:169-171) to beat state styles the same file authored (picker:194-203). The `.planner-detail-mode` rules `!important` four `display`s (planner:523-539) to override structure-coupled selectors. The autofill selector group even lists the **same selector twice** (picker:210 == 215). High-specificity authoring (`.modal.modifier .child`) raised the floor, so overrides now need `!important`, so the next override will too. This is a maintainability ratchet that only tightens.

### 4. ~95 lines of self-declared-dead `.debug-*` CSS, fully re-themed — Finding #8
`workout-base.css:948-1043` is gated by a comment saying it's "no longer used," yet it still carries a `z-index:50` (colliding with the live planner overlay), hardcoded hex, and a complete `@media`-dark + `theme-dark` + `theme-light` triplicate (L1002-1043) that someone *maintained* for a dead element. It is dead weight that actively costs maintenance every time the theming pattern changes, and it pads the hex/z-index counts that mask real problems.

### 5. `.status-overlay` double-defines its fade, and overlays are an unscaled z-index pile — Finding #4, #9, #14
`.status-overlay` (base:1055-1058) sets `opacity` and `transition` twice in one rule — two fade durations (0.2s vs 0.3s) where only the second is live, on the exact property (the countdown fade) the rule exists to control. It sits in a pile of seven `inset:0` fixed overlays (settings/picker/welcome/status/chart-empty) whose stacking order is bare magic ints (40/45/46/50/60/61) with no scale, no variable, and inconsistent scrims — adding or reordering any modal is grep-the-whole-tree-and-guess.

### Honorable mentions (low severity, real)
- `--hover-medium` and `--hover-light` are byte-identical in dark mode (base:127-128) — the 3-step intensity scale silently degrades to 2 steps, so the names mislead (#12).
- Empty rule `.planner-day.has-month-label` (planner:179-181, comment only) and no-op `.wb-chart-axis-overlay--grid` (picker:958-960, re-sets inherited `z-index:1`) — clutter implying structure that isn't there (#11).
- `.picker-search-wrap input[type="search"]` declared twice (picker:117 and 266) — edits can land in the dead copy (#11).
- `welcome.css:12` `var(--bg, #0c0c0c)` — the only hex outside the token file is a fallback that doesn't match either theme's `--bg` (`#f4f4f4`/`#222222`), so if the var ever fails to load the overlay flashes near-black in light mode.
